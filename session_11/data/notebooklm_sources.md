## Notebook 1: CCI Session 11 - Lesson 1: What Is a Web Page? HTML from Scratch

A web page is just a text file that the browser knows how to draw. The language it is written in is HTML — HyperText Markup Language — which marks up content with structure the same way a patient chart has a structure of headings, sections, lists, and paragraphs. HTML works through tags: a word in angle brackets, usually in pairs, an opening tag and a closing tag with a slash, wrapping some content. The grammar is always open, content, close. A handful of tags cover almost everything: h1 through h6 for headings, p for paragraphs, ul and li for bulleted lists, a for links, strong for bold, and br for a line break. Tags can carry attributes — settings written as name equals value inside the opening tag — for example the href attribute on a link tells it where to go. Every page shares a standard skeleton: a DOCTYPE html declaration on the first line, then an html element containing a head (information about the page that the visitor does not see, like the title that shows on the browser tab) and a body (everything the visitor actually sees). You build a web page with nothing but a text editor and a browser: type the skeleton, save the file with a .html ending, and double-click it to open it in the browser. There is no server and no internet required. The edit-save-refresh loop — change the file, save, refresh the browser — is the heartbeat of all frontend work. A form is built from a form tag wrapping input fields and a button; the input type attribute changes what it accepts, text or number. But a plain HTML form does nothing when clicked — it has structure but no behavior, because HTML describes structure only. A small amount of CSS, the language of appearance (fonts, colors, spacing), keeps a page from looking dated; HTML is structure, CSS is looks. The most common beginner mistake is forgetting a closing tag, which produces no error message — just a page that looks wrong — so when a page renders strangely the first move is to find the tag you opened and never closed.

## Notebook 2: CCI Session 11 - Lesson 2: Making It Move — JavaScript and the Frontend Landscape

HTML is a noun language: it names things but has no verbs. To make a page act — respond to a click, check a value, show a warning — you need JavaScript, the one programming language that runs natively inside every browser with nothing to install. The three frontend languages divide cleanly: HTML is structure, CSS is appearance, JavaScript is behavior. JavaScript lives in a script tag. A function is a named block of instructions; an event such as a click, wired with an attribute like onclick, triggers the function. Inside, document.getElementById finds a specific tag by its id attribute, .value reads what the user typed, and the code can write back into the page by setting an element's textContent. That read-something, compute-something, write-something-back cycle is what every interactive page does. A clinically flavored example reads a temperature input, uses parseFloat to turn the typed text into a number, and an if/else to show a febrile warning above 38 degrees. A critical caveat: this check runs entirely in the browser on the user's own machine, which is fine for a convenience warning but is not where clinical logic belongs for anything that matters, because anyone can open the browser tools and change it. Real decision rules run on the backend, on a server the user cannot reach into. The frontend framework landscape — React, Vue, Angular, Svelte — exists to manage scale. When a screen has hundreds of fields that must stay in sync with the underlying data, manual getElementById updates become a tangle. Frameworks share one big idea: you describe what the screen should look like for a given set of data, and the framework keeps screen and data in sync for you. React, backed by Meta, is the dominant one and builds UIs from reusable components; Vue is gentler; Angular, from Google, is big and opinionated; Svelte does its work at build time. They are more alike than different. This course deliberately does not use a framework for its clinical apps, because the ER-triage app has a form and a queue, not hundreds of live-syncing fields — Django renders the HTML on the server and a little plain JavaScript handles in-page behavior, with no build step. Reaching for a framework is a judgment: pay its complexity only when the screen's complexity justifies it. The course's quizzes and practices are written in React, so a .jsx file is a React file. JavaScript is case-sensitive and unforgiving; when a button does nothing, the browser's developer console shows the error in red.

## Notebook 3: CCI Session 11 - Lesson 3: The Full Stack — Frontend, Backend, and Where Django Lives

Everything that runs in the browser is the frontend, and it has a hard limit: close the tab and the data is gone, and anything sent to the browser can be read by anyone. So data and trusted logic live on the backend — a server, a computer that is always on, holding the database and the logic you control. Together the browser half and the server half are the full stack. The two halves communicate through the request/response cycle: the browser sends a request, the server receives it and does its work (validate the data, run a rule, save a row, call an LLM), then sends back a response (a finished HTML page or a redirect), and the browser draws it. Every action on every website is some version of this loop. In the ER-triage app, a nurse submits the triage form (a request carrying the patient data); the server validates it, computes the acuity, asks the LLM about oncologic emergencies, saves the row, and renders the confirmation page (the response). The acuity is computed on the server precisely because clinical logic the user must not be able to rewrite belongs on the backend. This frontend/backend split is one of the oldest patterns in software. In the 1990s and 2000s the dominant approach had the server assemble the HTML page on each request and send the finished page to the browser; Microsoft called their version Active Server Pages (ASP), and the PHP language did the same and still powers much of the web. Django is this same pattern — server-side rendering — done well in Python: a request arrives, Django runs your Python, pulls data from the database, drops it into an HTML template, and sends the finished page back. It contrasts with client-rendered single-page apps (a React app where the browser builds the screen from raw data), but for clinical internal tools server rendering is simpler, faster to ship, and easier to secure. Django is a backend web framework that hands you pre-built the parts every data-driven site needs: models (data definitions and a database), a URL router (mapping URLs to Python functions), views (logic that runs per request), templates (HTML with data slotted in), plus a free admin, user accounts, and security. The single organizing sentence: Django is the backend — it receives the request, runs Python, reads and writes the database, and renders the HTML page that goes back to the browser. Deployment, later, simply means putting that backend and database on an always-on, internet-reachable computer. The two most expensive clinical mistakes are putting backend work in the frontend: trusting the browser (validation or rules that run only in JavaScript can be bypassed, so anything that must be true is enforced on the backend) and leaking a secret (an API key or another patient's PHI that reaches the browser is exposed).

## Notebook 4: CCI Session 11 - Lesson 4: The Smallest Django That Runs

Django distinguishes a project from an app and the two must not be confused: a project is the whole website with its settings and database config; an app is one feature inside the project, such as triage, billing, or dashboard. One project contains many apps. Setup follows the same hygiene as the IDE session: a folder, a virtual environment created with python -m venv and activated, then pip install django. Two commands create the skeleton: django-admin startproject mysite . (the trailing dot means create it here) makes the project, and python manage.py startapp pages makes an app. The resulting files each have a job: manage.py is the control panel you run every command through; inside the project folder, settings.py holds all configuration including INSTALLED_APPS and the database, urls.py is the master URL map, and wsgi.py is the entry point a real server uses; inside the app, views.py holds the Python that runs on a request and models.py holds the data definitions. Running python manage.py migrate sets up the starter database and python manage.py runserver starts a small web server on your machine at 127.0.0.1:8000, also called localhost, meaning this very computer. Serving a page takes three moves that every Django page needs: first register the app by adding its name to INSTALLED_APPS in settings.py (the step beginners forget); second write a view, a Python function that takes a request and returns a response; third connect a URL in urls.py with a path entry mapping a URL to that view, where the empty path is the homepage. The rhythm of all Django work is URL to view to response. Returning HTML as a raw Python string is ugly; the right way is a template — an HTML file with double-brace slots like name that the backend fills in — rendered with the render shortcut, which passes a dictionary of values. This is server-side rendering: the data is decided by Python on the server and baked into the HTML before it is sent. Templates live in a doubled folder, app/templates/app/, so that templates from different apps with the same name do not collide. Common snags: the server may need a manual restart to pick up a change; TemplateDoesNotExist almost always means the doubled-folder path is wrong or the app is not in INSTALLED_APPS; and a new terminal requires re-activating the virtual environment, signaled by the (.venv) in the prompt.

## Notebook 5: CCI Session 11 - Lesson 5: Meet the ER-Triage App

The ER Triage Extractor is the runnable clinical Django app the course designed in Session 10 and builds on here: a triage nurse submits a patient's vitals and chief complaint, the system assigns an acuity from 1 to 5 and flags suspected oncologic emergencies, and the on-call doctor sees a live queue sorted by acuity. The goal of this lesson is the skill of reading an existing codebase — far more common in real work than writing from scratch — meaning the ability to find your way around and know which file to open to change a given thing, not to understand every line. You run it the same way as any Django app: a virtual environment, pip install -r requirements.txt (the package list), migrate, a custom seed_demo command that adds five fake patients, and runserver. The doctor queue is at the root URL, the nurse form at /triage/new, and a detail view at /triage/<id>. Without an OpenAI key the app still runs because the emergency-flag step degrades gracefully, saving triages with no flags and a status of failed rather than crashing; adding a key in a .env file enables real flags. Reading the project first: settings.py has INSTALLED_APPS listing the triage app and the database setting (SQLite, a database in a single file, ideal for development); the project urls.py uses include to delegate all triage URLs to the app's own urls.py, which keeps projects organized. Inside the triage app, urls.py maps four URLs to four views, each given a name that templates link by, with slots like <int:event_id> capturing part of the URL. models.py defines two models — Patient (MRN stored encoded, name stored encrypted, via crypto.py using Optimus for the MRN and Fernet for the name, so raw PHI is never in the clear) and TriageEvent (the complaint, vitals, oncology context, acuity, and flags). The services folder holds the brains and its split is the key design idea: acuity.py is pure Python arithmetic with no LLM and no network, because acuity is a number in the record that must be deterministic and testable; oncologic_emergency.py is the LLM call, which only accepts answers from a closed set of four emergency labels and returns failed on any error. The architecture itself enforces the clinical-safety rule that the LLM never computes acuity. forms.py validates every field's range on the backend; views.py contains the logic, where nurse_form_view on a POST validates, computes acuity, calls the extractor, upserts the patient, creates the event, and redirects. Templates live in triage/templates/triage/, with base.html the shared frame and a for-loop in doctor_queue.html building the table. The reading technique is to start from what the user does, find the URL, follow it to the view, and open only the files that view uses — letting the request be your guide.

## Notebook 6: CCI Session 11 - Lesson 6: Extend It — Add Your Own Django App

The most realistic software task is extending a working application rather than building from scratch: someone says the app is great, can it also show me X, and you make it so. Here X is a dashboard — the doctor queue shows individual patients, but the unit manager wants the shape of the queue, how many patients at each acuity level and how many emergencies flagged. Because the dashboard is conceptually its own thing, a different view onto the same data, it earns its own Django app. The work always starts from a running, working state so that any break is known to be caused by your change. The five-step loop for adding any feature: first python manage.py startapp dashboard creates the app folder on disk, but the project does not know about it yet; second add the app's name to INSTALLED_APPS in settings.py, the single line that connects the new app to the project and the first place to look if the app mysteriously does nothing; third write the view in dashboard/views.py, which imports TriageEvent from the triage app (apps in the same project can use each other's models freely, sharing one database) and counts events by acuity and by emergency flag, creating no new data or model; fourth write the template at dashboard/templates/dashboard/dashboard.html, using the doubled-folder pattern and extending the triage app's base.html to inherit the header and styling, with extends and block to share the frame, a for-loop over the acuity counts, and the widthratio helper to size bars; fifth connect the URL by giving the dashboard app its own urls.py and adding an include line in the project urls.py so /dashboard/ routes into it. Then runserver and open /dashboard/ to see it counting the seeded patients. A link is added with the url template tag, which looks up a route by its name rather than hardcoding the path. The named loop — startapp, INSTALLED_APPS, view, template, url — is the entire rhythm of extending a Django application, identical for a feature that writes data except you would also add a model and a form. Two failure modes catch everyone the first time: forgetting INSTALLED_APPS, which makes Django ignore the app entirely, and the doubled template folder, where the file must be at app/templates/app/file.html. Both errors are loud and fixed in one line, and a Django change that does nothing is almost always one of these two.

## Notebook 7: CCI Session 11 - Lesson 7: Ship It — GitHub to Render to a Live URL

The address 127.0.0.1:8000 means this computer only; deployment moves the backend onto a computer that is always on and reachable from the internet, and the request/response cycle does not change, only the address — localhost becomes your-app.onrender.com. The path has two stops: GitHub, where the code goes up to a cloud repository, then Render, a hosting service that watches the repository, runs the app, and gives it a public address. Render is used instead of Azure on purpose: it connects straight to a GitHub repo, has a free tier, and turns deployment into a few clicks, while the concepts — a build step, a start command, environment variables for secrets, static files, a database — are identical everywhere. Three files a server needs are already in the app: requirements.txt lists the exact packages the server installs (including gunicorn and whitenoise); build.sh is the build step run on every deploy, which pip installs the packages, runs collectstatic to gather CSS and JS into one folder for whitenoise to serve in production, and runs migrate to set up the database; render.yaml is the blueprint telling Render to use Python, run build.sh to build, run gunicorn er_triage.wsgi to start (gunicorn being the production web server that replaces the development-only runserver), attach a small persistent disk for the database, and which environment variables to set. Three production settings, all wired to read from the environment: DEBUG must be False in production because its helpful error pages leak internal details; ALLOWED_HOSTS lists the hostnames the app answers to, with Render's host added automatically; and SECRET_KEY, a cryptographic secret, is generated by Render rather than written in code. Step one pushes to GitHub with git init, add, commit, then git remote add and git push; crucially the app's .gitignore excludes .env, db.sqlite3, and the virtual environment, because a committed secret is a leaked secret that stays in the history even if deleted. Step two creates the Render web service by connecting the repo, which Render reads via the blueprint, on the free tier. Step three sets the secrets marked sync false in the dashboard so they never live in the repo: OPENAI_API_KEY (optional, the app degrades without it) and FERNET_KEY, generated locally. Step four watches the deploy, where a first failure is normal and the log explains it — common causes are ALLOWED_HOSTS, a missing package, or build.sh not being executable; when green, Render shows the live URL. The deploy loop is push to deploy: change something, git push, and Render redeploys automatically at the same URL. A clinical-looking app on the public internet is right for the course with fake data labeled not for clinical use, but a publicly reachable URL with real patient data is a different universe needing authentication, PHI controls, audit logging, and governance — the subject of Session 12.

## Notebook 8: CCI Session 11 - Lesson 8: Cheat Sheet and What to Build Next

The session crossed the entire web stack, from being unable to write a line of HTML to a clinical web application live on the internet that the student built, extended, and deployed. The working developer's skill is recognition, not recall — knowing a thing exists and roughly where it goes, so you know what to look up when you build. The whole stack on one page: frontend in the browser is HTML for structure, CSS for appearance, JavaScript for behavior, and frameworks like React for complex UIs; backend on the server is Python and Django following URL to view to template, with models, forms, and services, and with deterministic logic and secrets kept here; the database holds rows and tables like Patient and TriageEvent. The organizing rule is that display and interaction go in the frontend while data and trusted logic go in the backend — validate on the backend, keep secrets on the backend, never let the browser be the law. The Django command set covers startproject, startapp, runserver, makemigrations, migrate, createsuperuser, and test. The file map ties each file to its job. The loop for adding any feature is startapp, register in INSTALLED_APPS, write the view, write the template, wire the URL. Template syntax includes the double-brace variable slot, the for loop, the if branch, the url tag that links by route name, and extends with block for inheriting a shared frame. The Render deploy checklist: requirements.txt complete, .gitignore excluding secrets, build.sh installing and collecting static and migrating, the gunicorn start command, DEBUG false, ALLOWED_HOSTS including the host, the secrets set as environment variables, and git push to auto-deploy. What to build next, in rising ambition: harden this app by adding the authentication it deliberately lacks and moving to Render Postgres; build your own one-screen tool the unit actually needs; or connect the chain by using Claude Code's agents to build the next feature through the Software Factory pipeline and deploying it. In the course arc, Session 10 built software, Session 11 shipped it, and Session 12 makes it safe — PHI de-identification, audit logging, API-key hygiene, and the governance that stands between a teaching demo and a tool that touches a real patient. A clinical AI developer needs both the excitement of a live URL and the discipline of governance.
