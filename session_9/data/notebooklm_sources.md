## Notebook 1: CCI Session 9 - Lesson 1: VS Code Setup

VS Code is a free lightweight code editor from Microsoft, and unlike notebooks it is built around a workspace which is just a folder you opened with File Open Folder. Every real clinical AI system at KHCC lives in a folder of Python files on a server, not in scattered notebook cells. Setting up that folder properly is the foundation of professional development. The integrated terminal in VS Code is the same shell as your system terminal, just docked inside the editor, opened with Ctrl backtick. Virtual environments (venv) isolate each project's Python packages so they do not collide across projects. The setup sequence is: open the project folder, open the terminal, create a venv with python -m venv .venv, activate it (source .venv/bin/activate on Mac and Linux, .venv backslash Scripts backslash activate on Windows), then the single most-missed step — click the Python version in the bottom-right of VS Code and select the venv interpreter. Without this step every pip install lands in the wrong Python. After the venv is selected, create requirements.txt listing dependencies (gradio and python-dotenv for our calculator app), run pip install -r requirements.txt, and create a README.md explaining how to run the project. Run which python before and after activation to see that the venv changes which Python the terminal uses. If you skip the venv every project's packages pile into the same global Python and eventually nothing works.

## Notebook 2: CCI Session 9 - Lesson 2: gitignore and the Secrets Mindset

The most important 15 minutes of local development is learning what NOT to share. Every week clinicians somewhere leak an API key or a patient list to GitHub, and once a secret is pushed it lives forever even after you delete the commit. Git tracks every file in your folder by default. The dot gitignore file tells git to ignore specific patterns. Secrets like API keys, passwords, and PHI must never enter git history. The three-file pattern that prevents leaks: dot env holds your actual secret values and is excluded by gitignore so it is never committed. Dot env example shows the shape of which variables exist without their values and IS committed so a teammate cloning your repo knows what to set. The gitignore file itself is committed. Python reads secrets via the python-dotenv library: from dotenv import load_dotenv, load_dotenv(), then os.getenv("OPENAI_API_KEY"). The live failure mode: scrapers continuously search public GitHub for patterns like sk hyphen and harvest leaked OpenAI keys within minutes of being pushed. The correct response when you accidentally commit a key is to rotate it immediately at the provider — not to delete the commit, because the key is already in scraper logs. A complete clinical gitignore covers: dot venv slash, double underscore pycache double underscore slash, asterisk dot pyc, dot env, dot DS underscore Store, asterisk dot db, data slash. The mental rule for the rest of the course: if a file contains anything you would not print in a journal article, it goes in dot gitignore.

## Notebook 3: CCI Session 9 - Lesson 3: Git and GitHub from VS Code

Git tracks changes to files. GitHub is a website that stores git repos. They are not the same thing. The mental model is four stages: Working directory where you edit files, Staging area where files wait to be committed via git add, Local commit which is a snapshot of staged changes via git commit, and Remote which is GitHub via git push. A commit is a snapshot of the whole project at a point in time, not a file. The Source Control panel in VS Code (Ctrl Shift G) is just buttons that run git add, git commit, and git push under the hood. Initializing a repo: git init, git config user.name and user.email. Pre-flight check before the first commit: confirm dot venv does NOT appear in Changes — if it does, gitignore is wrong and must be fixed first. Connecting to GitHub: create a new repository on github.com without initializing a README, then git remote add origin URL, git branch -M main, git push -u origin main. Authentication options ranked by ease: HTTPS plus Personal Access Token (github.com Settings Developer settings Personal access tokens Generate new token classic, tick repo scope, paste token as password when prompted), GitHub CLI gh auth login (one command, handles everything), SSH key (cleaner long-term but more setup). Common pitfalls and fixes: Permission denied publickey means you're using SSH without a key set up — switch to HTTPS or run ssh-keygen. Remote already exists means you ran git remote add twice — use git remote set-url origin instead. Updates were rejected means someone pushed first — run git pull --rebase then push again. git status is the most-used command, always your first move when confused. The undo cheat sheet: git checkout -- filename discards unsaved changes, git restore --staged filename unstages, git reset --soft HEAD~1 undoes commit but keeps changes, git reset --hard HEAD~1 throws away last commit destructively, git commit --amend fixes the last commit message. Golden rule: if you've already pushed to a shared branch, don't rewrite history.

## Notebook 4: CCI Session 9 - Lesson 4: Build the Medical App with Gradio and SQLite

The shape of every real KHCC tool — Python logic plus a small database plus a UI — is exactly what we build in this lesson with a Cockcroft-Gault creatinine clearance calculator. Gradio turns a Python function into a web UI in about 10 lines of code, perfect for prototypes. SQLite is a database that lives in a single file on disk with no server to install or configure. The Cockcroft-Gault formula estimates creatinine clearance for renal dosing: CrCl equals open paren 140 minus age close paren times weight in kilograms divided by open paren 72 times serum creatinine close paren. Multiply the result by 0.85 for females. Build init_db.py first to create drugs.db with a nephrotoxic_drugs table containing name, class, crcl_threshold, and note columns, then insert 5 drugs: Vancomycin a glycopeptide with threshold 50, Gentamicin an aminoglycoside with threshold 60, Cisplatin a chemotherapy with threshold 60, Methotrexate an antimetabolite with threshold 60, Acyclovir an antiviral with threshold 50. Then build app.py with calculate_crcl that validates inputs are present, applies the formula with female correction, queries the database for drugs whose threshold exceeds the patient's CrCl, and returns a formatted alert string. The gr.Interface uses Number components for age, weight, and creatinine, a Radio component for sex, and two Textbox components for the CrCl result and drug alerts. The auto-init pattern is critical: at the top of app.py write if not os.path.exists("drugs.db"): import init_db so Hugging Face Spaces builds the database on first run. Test case: age 65, weight 70, creatinine 1.8, female yields CrCl approximately 26 mL/min with all 5 drugs flagged. After it works locally, git add specific files (NOT dot env or drugs.db which are gitignored), commit, push, and view the diff on GitHub.

## Notebook 5: CCI Session 9 - Lesson 5: Deploy to Hugging Face Space

A Hugging Face Space is a git repo that automatically deploys whatever code is in it, so your app goes from local to a public URL in minutes. The Gradio SDK tells Hugging Face to find app.py and run it. Secrets like API keys go in the Space's Settings Variables and secrets panel, never in your code — they are injected at runtime as environment variables that os.getenv reads exactly the same way as a local dot env. The free tier runs on CPU and sleeps after inactivity, which is perfect for teaching but not for production traffic. Spaces are public infrastructure: they are perfect for demos, never appropriate for PHI. Create the Space at huggingface.co New Space, name it crcl-calculator, choose Gradio SDK, MIT license, Public visibility. Clone the Space repo in a NEW terminal in a separate folder from your existing project to avoid tangling two git repos: cd to home, git clone the Space URL, cd into it. Authentication uses your Hugging Face username and an access token from huggingface.co Settings Tokens with WRITE scope. Copy app.py, init_db.py, and requirements.txt from your original project into the cloned Space folder. Replace the auto-generated README.md with one that has the required Hugging Face Spaces YAML header: title, emoji, colorFrom, colorTo, sdk gradio, sdk_version 4.44.0, app_file app.py, pinned false. Without this header Spaces will not build. Then git add dot, git commit, git push. Open the Space URL and watch the build log — within 1 to 2 minutes the app is live and shareable. To add secrets: Settings Variables and secrets, add OPENAI_API_KEY, redeploy. The key is now an env var inside the Space container, never visible in the repo, never exposed to viewers.
