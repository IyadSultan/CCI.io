## Notebook 1: CCI Session 12 - Lesson 1: What Is a Network, Really?

A network is almost embarrassingly simple: it is computers passing messages to one another, and the internet is networks of networks — every hallway in the world connected, agreeing to carry each other's notes. Picture a hospital: each room is a computer, the hallways are the network, and a runner carries a slip from one room to another. The room that asks is the client and the room that answers is the server — the request/response cycle you met in Session 11, where your browser asked and the server replied. For a note to find its room, the room needs an address: the IP address, like a number painted on a door. Two addresses matter most. 127.0.0.1, called localhost, means this computer only — a room with no door to the hallway, which is why your app at 127.0.0.1:8000 was reachable only from your laptop. A public URL like your-app.onrender.com points to a real public address — a room with a door open to the whole internet. But a computer does not have one door; it has thousands of numbered doors called ports, and a service inside listens on one of them. Four are worth memorising: port 80 is plain HTTP, 443 is encrypted HTTPS (the padlock), 22 is SSH for remote login, and 3389 is RDP for remote desktop. A full address is IP plus port, like 142.250.65.78:443 — which room, and which door. The firewall is the reception desk: it checks only which room and which door each note is trying to reach, then allows or blocks it per a rule list — it does not read the contents. The single most important control is deciding which doors may be reached, by whom, and shutting the rest. The hospital's own internal network is a LAN or intranet — the inside of the building — while the public internet is the street, and the strongest move for a sensitive system is to give it no street door at all. The sentence to carry through the whole session: reachable from anywhere is both the entire point of deployment and the entire danger, because the same open door that lets your users in lets attackers try the handle too. The discipline is to open the minimum: one locked door, 443, for the public, and everything else shut, because every open port is attack surface that automated scanners knock on all day, every day.

## Notebook 2: CCI Session 12 - Lesson 2: Encryption, Plainly

Encryption, at its simplest, is scrambling a message with a key so that only someone with the key can unscramble it; without the key the result is meaningless gibberish. It matters because a note crossing the hallway passes people standing in it. A man-in-the-middle is someone positioned on the path your messages travel — on shared Wi-Fi, a compromised router, anywhere in between — who can read or even alter what passes if it is not encrypted. The clearest analogy is the post: plain HTTP is a postcard that anyone who handles it can read along the way, while HTTPS is a sealed envelope whose contents are encrypted in transit. That is why your deployed Session 11 app was served at an https address: even though it held only fake data, sending real patient data over plain http would be a postcard passed hand to hand down the hallway. The browser padlock is widely misunderstood. It proves exactly one thing: that your connection to the site is encrypted, so nobody in between can read the traffic. It does not prove the site is honest, safe, or run by who you think — a scam site can hold a perfectly valid padlock. So encrypted does not mean trustworthy; it means private in transit. The padlock answers who can read this on the way, not should I trust the person at the other end. Anything that carries sensitive information across a network must be the sealed envelope, never the postcard, and the next lesson opens the envelope to show the two different kinds of keys that make it work and the difference between protecting data while it moves and protecting it while it sits.

## Notebook 3: CCI Session 12 - Lesson 3: Two Kinds of Keys

There are two families of encryption. Symmetric encryption uses a single shared secret key to both lock and unlock — it is fast, but both sides must somehow already share the same key. Asymmetric encryption uses a pair: a public key anyone may use to encrypt, and a private key that only its holder can use to decrypt. Asymmetric solves the hard problem symmetric leaves open — how to exchange a key with someone you have never met without an eavesdropper catching it. Running alongside this is a second distinction that is not about the kind of key but about when the data is protected. Encryption in transit guards data while it moves over the network — that is TLS and HTTPS, the sealed envelope from Lesson 2. Encryption at rest guards data while it sits stored on a disk or in a database. You want both, because a system can encrypt its traffic perfectly and still leave the stored database readable to anyone who steals the disk. Your Session 11 app showed both at once: it Fernet-encrypted the patient name before saving it — that is encryption at rest, and Fernet is symmetric — while HTTPS protected the same name in transit. The TLS handshake that begins every HTTPS connection is the two families working together: asymmetric encryption is used at the start to safely agree on a shared symmetric session key, then the fast symmetric key does the heavy lifting for the rest of the conversation. The names worth knowing are AES-256 for symmetric, RSA for asymmetric, and TLS 1.3 for the secure channel. And the iron rule: never roll your own crypto. Cryptography is extraordinarily easy to get subtly wrong in ways that look fine but are catastrophically broken, so you always use vetted standards that have survived decades of expert attack — which is exactly why the course used Fernet rather than inventing something.

## Notebook 4: CCI Session 12 - Lesson 4: How Attacks Actually Work

This lesson follows one attack from beginning to end, defensively, so you can recognise the shape and name the brake at each stage — never to learn technique. The chain usually runs like a story. It can start with a zero-day, an unknown software flaw, but far more often it rides an old vulnerability that was patched months ago and never updated. An exploit turns that flaw into a foothold. The attacker installs a backdoor or remote-access tool, then establishes persistence so the access survives a reboot. From the first machine they perform lateral movement, spreading sideways to hunt higher-value targets like the electronic health record or the backups. They quietly carry out data exfiltration, copying data out of the building, and then weaponise encryption — locking the victim's own files — and demand payment: ransomware and extortion. The uncomfortable truth is that the most common way in is not a brilliant exploit; it is a person clicking a phishing link or typing a password into the wrong box, combined with a hole nobody patched. That is why the defenses are mundane and powerful. Patching closes the known holes attackers reach for first. Least privilege shrinks the blast radius, so a compromised account can only do what that limited account was allowed to do. Network segmentation divides the network into compartments so a breach in one cannot freely roam into the rest, braking lateral movement. And tested, offline backups — copies the attacker could not also encrypt — are what let a hospital restore its systems and refuse to pay the ransom. Each defense breaks the chain at a different link, which is why no single wall is enough; you assume one will fail and make sure the next one holds.

## Notebook 5: CCI Session 12 - Lesson 5: Defending Yourself — Passwords, MFA & Least Privilege

The highest-value thing you can do personally is three habits that need no code and work tonight. First, length beats complexity. The old advice to add a capital, a number, and a symbol produced passwords like P-at-ssw0rd that are annoying to type, easy to forget, and not actually strong, because attackers' tools try every such substitution first. What truly resists guessing is length, and the human-friendly form of long is a passphrase: four or five random, unrelated words, which is both stronger and easier to remember. Second, never reuse a password. When some old site is breached, attackers immediately try that same email-and-password pair on your email, your bank, your hospital portal — reuse turns one leak into a master key. Since no one can remember a hundred unique passphrases, a password manager does it for you: you memorise one strong master passphrase and it generates, stores, and fills a unique random password for every site. Third, multi-factor authentication. Even a perfect password can be phished or breached, so MFA requires evidence from at least two different categories — something you know (the password), something you have (your phone or a hardware key), something you are (a fingerprint). Your KHCC login plus a code on your phone is exactly this, and that second prompt is the wall that keeps a stolen password from becoming a stolen account. Not all MFA is equal: the ranking is SMS below an authenticator app below a hardware key or passkey, because SMS can be intercepted or stolen by SIM-swap, while a hardware key is phishing-resistant by design — it cryptographically checks the real site before responding, so a fake login page gets nothing. Watch for MFA fatigue, where an attacker who has your password floods your phone with approval prompts hoping you tap approve to make it stop; never approve a prompt you did not personally start. Finally, least privilege is a daily habit, not just an IT policy: do everyday work in a normal account, reach for admin rights only when truly needed, and request only the access you require, so that if your account is compromised the blast radius stays small. The same two ideas — a strong key instead of a guessable password, and minimum privilege — return in Lesson 9 as the SSH-key, non-root server.

## Notebook 6: CCI Session 12 - Lesson 6: The Honeypot Trick — Canary Tokens

Lessons 4 and 5 were about prevention, but the hard truth every security team learns is that you cannot prevent every breach, so mature security adds a second layer: detection. If you cannot stop everyone from getting in, you can at least know the instant someone does. A honeypot is bait, and a canary token is the smallest, sharpest kind — a deliberately planted decoy, usually a document or a link, that has no real purpose except to alert you the moment someone interacts with it. The name comes from the coal mines, where a caged canary collapsed from invisible gas long before the miners did, buying time to escape. You do not build any of this: a company called Thinkst runs canarytokens.org, completely free, with no login. You generate a token, enter your own email address (where the alert is sent) and a short memo reminding you where you planted it, download the file, and place it somewhere a snoop would find tempting. A vivid clinical example is a Word document named something irresistible like Patient_List_2026.docx, containing nothing real, dropped in a shared department folder; the day someone opens it, it quietly phones home and you get an email — typically with the date, time, and source IP of whoever opened it. Read that alert carefully and hold two columns in your head: it tells you when the decoy was touched, a real and precise event, and roughly where the request came from, but the IP can be a VPN, a proxy, or a shared office connection, so it is a signal, not an identity — a strong reason to investigate, not courtroom proof of who sat at the keyboard. The ethics line must stay bright: a canary is detection on your own assets, a burglar alarm in your own house. Use your own email, keep the bait empty and fake with no real PHI ever, plant only where you have authority and coordinate with IT and security first, and place it where touches are rare and meaningful — a token on a busy folder fires constantly and you learn to ignore it.

## Notebook 7: CCI Session 12 - Lesson 7: PHI and HIPAA — The Rules of the Road

Health information becomes Protected Health Information when two ingredients are present together: health information — a diagnosis, a lab value, a medication — plus an identifier that ties it to a specific person. Either alone is usually harmless; a febrile-neutropenia note with no name could be anyone, and a name in a phone book is just a name, but put them together and you have PHI and a duty of care. HIPAA is the United States law that defines this, and although Jordanian law and KHCC policy are what bind us legally, HIPAA's definitions have become the international common language, so the AI Office adopts its principles as a floor. HIPAA has two halves: the Privacy Rule governs who may use or share PHI, and the Security Rule governs how you must technically protect it, where encryption, access control, and audit logs live. HIPAA names eighteen kinds of identifier — names, geographic detail smaller than a state, dates finer than a year, phone and fax numbers, email, national ID, medical record numbers, account and beneficiary numbers, license and vehicle and device identifiers, URLs, IP addresses, biometrics, full-face photos, and any other unique characteristic — and stripping all of them is the Safe Harbor method of de-identification, with two riders: dates reduce to the year and ages over 89 bundle into a single ninety-plus category. Three jobs are constantly confused and are not interchangeable: de-identification removes or transforms the identifiers so the data is no longer PHI at all; encryption scrambles the data so it cannot be read without a key, but encrypted PHI is still PHI — you have locked the cabinet, not emptied it; and access control decides who is allowed in. You generally want all three, and your Session 11 app deliberately used two — it encrypted the name and encoded the MRN but had no login, which is exactly why it was labelled not for clinical use. Two principles cause most everyday violations: minimum necessary, meaning use and share only the least PHI the task requires, and the trap of it's just for research, which has its own stricter governance. The genuinely hard problem is finding PHI in free text, where a rules pass catches the shaped identifiers and an AI pass catches the subtle ones, but a scanner is a power tool for a human reviewer, never a replacement — and the deepest danger is re-identification, where a rare diagnosis plus an exact date plus a small town can point to one person with no name attached at all.

## Notebook 8: CCI Session 12 - Lesson 8: The Synthetic-Data Pipeline — Safe Data by Design

If real PHI is this hard to protect and easy to leak, the AI Office's answer is to stop keeping it around for demos, testing, and learning: the safest patient data is data that was never real. There are two postures toward PHI. The reactive posture lets real data flow wherever the work takes it and then tries to protect every copy — exhausting, and it always loses because you only have to slip once. The preventive posture, which the office calls safe data by design or preventive de-identification, arranges things so the dangerous data never leaves its secure place, and everything downstream works on data made safe before it ever moved. Four rules are wired into the pipelines so the easy path is the safe one: every MRN is encoded before it is stored or shown, using the same encode_mrn function from your Session 11 app; every patient name is encrypted before any output, using the same encrypt_name function; the keys live in a managed secret vault, never in code; and models are evaluated on a frozen, governed, access-controlled cohort of deceased patients so honest performance numbers do not put living patients at risk. Most of the time you do not need real patients at all — you need realistic-shaped data, and that is synthetic data: a generated record that looks and behaves like a real one but belongs to no living person, so there is nothing to re-identify and nothing to leak. The kit ships two tools that prove each other. The PHI scanner reads a note and flags identifiers with a rules pass plus an optional AI pass; the synthetic generator mints fake records and applies the encode and encrypt rules before writing anything. Run the scanner on a real-style note and it lights up; run the same scanner on the generator's output and you get zero findings — not because the scanner is lazy but because the generator put nothing identifying into the output. In a real pipeline, real PHI appears in only two tightly fenced places, secure ingest and governed evaluation, and everything a human touches day to day is synthetic. The traps are leaky generators that lightly edit real records — swap the name but keep the real MRN, date, and rare diagnosis — which are re-identifiable and not genuinely synthetic, and the excuse it's only a test, which quietly rebuilds the reactive posture you were trying to escape.

## Notebook 9: CCI Session 12 - Lesson 9: The Cloud, Hardened — Azure CLI from Resource Group to Teardown

This lesson operates everything the session built by standing up a real server in Microsoft Azure and then deleting it. The cloud is the best teacher of infrastructure security because it is the one place you build the network, the firewall, and the server yourself and see exactly which doors you opened. It is done through Azure Cloud Shell, a terminal that runs in your browser with the az command-line tool already installed and logged in, so nothing is installed on your laptop; the instructor demonstrates live and you keep a numbered runbook to replay later. The lifecycle has a beginning, a middle, and the part everyone forgets, an end. First, a resource group is a labelled box that holds everything, and because every piece lives inside it, teardown is a single command. Second, create a hardened VM, with three choices baked into one command: a current patched Ubuntu image, a non-root admin user (least privilege), and generate-ssh-keys, which means you log in with an SSH key — asymmetric encryption doing authentication, where the private half never leaves your machine and cannot be typed, intercepted, or phished — rather than a password. Third, lock the Network Security Group, which is Lesson 1's reception desk written down: it denies by default, and you open the minimum — 443 to anyone because it is a public website, and 22 (SSH) only to your own IP address, never to 0.0.0.0/0, which is the number-one cloud breach. Fourth, install nginx and serve one page, using sudo, the controlled way a least-privilege user borrows admin rights for one command (and remembering that real clinical traffic must be HTTPS on 443 with a certificate, not plain HTTP). Fifth, verify from outside: confirm the page loads and audit the rule list so the closed doors really are closed. Finally, the step that separates a professional from an expensive accident: delete everything, because deleting the resource group erases the VM, its disk, network, NSG, and public IP in one sweep — a running VM is two liabilities at once, a bill that never sleeps and a 24/7 attack surface, and the most secure server is one that no longer exists. The last habit is to paste your configuration into an AI assistant and ask what is still exposed and what an attacker would try first — the cheapest security review you will ever run, a relentless second pair of eyes that never tires of reading rule lists.
