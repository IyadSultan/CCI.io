## Notebook 1: CCI Session 10 - Lesson 1: Writing a PRD with the PRD-Builder Skill

A PRD is a Product Requirements Document. It is the one-page-plus structured spec that turns a sentence like "we need a tool that flags missed AKI" into something the rest of the team — clinicians, engineers, data scientists, validators — can react to. Dr. Iyad Sultan wrote a Claude Code skill called PRD-Builder that takes a free-form description of a clinical AI idea and produces a complete PRD, asking targeted follow-up questions only about genuine gaps in the input. The skill works in three steps. Step one is classification: it decides whether the input describes an AIDI clinical product (one that lives at KHCC and touches VistA, Databricks, gpt-4.1-mini, MRNs, oncology workflows) or a generic software product. Step two is the checkpoint: the skill scans the input for five critical sections, marks each as answered, partial, or missing, and asks one focused batched question covering only the real gaps. Step three is the draft: it loads the matching template (AIDI clinical or generic), fills it in, marks inferred-but-unconfirmed assumptions as "[assumed — confirm]" and genuine gaps as "[TBD]" surfaced under an Open Questions section. The AIDI clinical PRD template has twelve sections including Clinical Context, Goals and Non-goals, Users and Stakeholders, Functional and Non-functional Requirements, Data Sources, Architecture, Validation Plan, and Open Questions. The generic PRD template covers TL;DR, Background, Goals, Non-goals, Personas, User Experience, Functional and Non-functional Requirements, Technical Considerations, Dependencies, and Success Metrics. The discipline is to always checkpoint before drafting (never draft on the first turn) but to ask only about real gaps, not nice-to-haves — those become [TBD] markers in the draft. The output is Markdown by default; Word format is produced only on explicit request. Length aims for completeness, not brevity — a PRD missing sections is worse than one with [TBD]s because the structure itself is the value.

## Notebook 2: CCI Session 10 - Lesson 2: Memory with CLAUDE.md

CLAUDE.md is a Markdown file at the project root that Claude reads before every session begins. It is the persistent project instruction manual — the equivalent of the order-set template the pharmacy hands every new resident, "this is how we do things on this service." Three levels of CLAUDE.md exist. Personal lives at `~/.claude/CLAUDE.md` and applies to every project on your machine. Project lives at `./CLAUDE.md` at the root of the codebase and is checked into git so the whole team gets the same rules. Directory lives in any subfolder of a project, and applies only when Claude is working in that subfolder. The compliance budget is roughly 150 to 200 instructions before Claude starts to drop adherence — so every line earns its place via the litmus test: "Would Claude make a mistake without this line?" If the answer is no, delete it. Beginners overfill CLAUDE.md with style preferences and aspirational rules; experienced users keep it tight. Tools for keeping it lean include @imports (reference another markdown file rather than pasting the content inline — Claude reads the imported file only when needed) and conditional rules (place rules in `.claude/rules/<name>.md` with path frontmatter so they only load when relevant files are in scope). After every mistake, the discipline is to ask "Update CLAUDE.md so this doesn't happen again." Over six months this auto-updating turns a starter file into a sharp one. The starter CLAUDE.md for clinical AI work at KHCC has six sections: stack (one-sentence summary of the tech), run and test commands, behavior rules, PHI rules (MRNs Optimus-encoded, names Fernet-encrypted, eval cohort frozen to deceased patients), and tool quirks (use `pytest` not `python -m pytest`, prefer `gh` CLI over GitHub MCP, never skip pre-commit hooks). Around sixty lines, every line justified.

## Notebook 3: CCI Session 10 - Lesson 3: Skills

A Claude Code skill is a packaged playbook that Claude reads on demand. Skills live in `.claude/skills/<name>/SKILL.md`. The medical analogy is a clinical pathway — for febrile neutropenia, you don't reinvent the workup at the bedside, you follow the pathway. Skills work the same way: when Claude detects a request that matches a skill's description, it reads the SKILL.md and follows the playbook. The difference from CLAUDE.md is critical. CLAUDE.md loads always; it carries persistent rules. Skills load only when invoked by description match; they carry capabilities. SKILL.md format is YAML frontmatter (name and description) followed by Markdown body describing the workflow. The description is the trigger — Claude scans skill descriptions at startup and decides which skill to invoke based on what the user asks for. Description should be written as "Use this skill when…" with concrete trigger phrases. A skill folder can also contain `references/` (background documents Claude reads when needed), `templates/` (file skeletons Claude can copy from), and `scripts/` (Python or shell helpers Claude runs). Supporting files load only when the skill is invoked, so they don't bloat the baseline context. The PRD-Builder skill from Lesson 1 is a good example to study — it has a SKILL.md describing the three-step workflow, two reference templates (AIDI clinical and generic), and the discipline of always checkpointing before drafting. Three KHCC skill examples worth building: an `aidi-extractions` skill that codifies the SQL gotchas of VistA tables and the PydanticAI patterns for structured extraction; an `r-clinical-analysis` skill that encodes the tidymodels + finalfit + survminer workflow for Cox regression and Kaplan-Meier curves with Lancet theming; a `scientific-writing` skill in Dr. Sultan's voice for IMRAD-disciplined manuscript drafting. Output styles are a separate, lighter feature — `/config` lets students pick Explanatory, Concise, or Technical voice for Claude's responses, independent of any skill.

## Notebook 4: CCI Session 10 - Lesson 4: MCP Servers and Anthropic Connectors

MCP stands for Model Context Protocol — the open standard for letting Claude connect to external services like Slack, Gmail, Postgres, Notion, Linear, Figma, monitoring dashboards, and any custom API. The medical analogy is the EMR's interface engine: VistA talks to the lab system and the pharmacy through standardized interfaces, and MCP is the same thing for Claude. Two flavors exist in the wild. Anthropic Connectors are productized integrations available in Claude.ai and Claude Desktop. They install with a click, ask for OAuth scopes, and are maintained by Anthropic. The roster includes Slack, Gmail, Google Drive, Google Calendar, Notion, Linear, GitHub, and a growing list. Custom MCP servers are anything you (or anyone else) writes, distributed by GitHub, npm, or PyPI. You add one to Claude Code with `claude mcp add <name> <command>`. Both speak the same protocol; the difference is curation, ease of install, and trust. Three install scopes exist. Local scope adds the MCP only to the current folder. User scope adds it for every project on your machine. Project scope writes the configuration to `.mcp.json` checked into git so every teammate inherits the same connection. For team-wide databases like the AI Office's `AIDI-DB`, project scope is correct. For personal connections like your own Notion workspace, user scope is correct. Once an MCP is connected, Claude can implement features from issue trackers ("add the feature in JIRA ENG-4521 and open a PR"), analyze monitoring ("check Sentry for the spike on the triage page"), query databases ("find ten random users who triggered the new alert"), and automate workflows ("draft Gmail invites to those users"). The single most important rule for clinical MCP servers is: read-only by default. Your AIDI-DB MCP should refuse anything but SELECT statements. The day someone writes "delete all rows where x" in a prompt is the day you learn this the hard way. For git, prefer the `gh` CLI over a GitHub MCP — CLI tools are more context-efficient than MCP servers for high-frequency operations.

## Notebook 5: CCI Session 10 - Lesson 5: Subagents and Agent Teams

A Claude Code subagent is a specialist with its own context window, system prompt, tool restrictions, and optional model assignment. The medical analogy is a subspecialty consult — narrow expertise, fresh chart, returns a written note. Subagents live in `.claude/agents/<name>.md` with YAML frontmatter defining four key fields: name (the identifier Claude uses to invoke the subagent), description (Claude scans this to decide when to delegate), tools (a whitelist of which built-in tools this subagent can use), and model (haiku for cheap read-only work, sonnet for reasoning). Three reasons to use a subagent. First, context hygiene — keep the main session focused on the high-level task while the subagent's separate context burns through file reads and grep. Second, focused expertise — a security reviewer with a tight system prompt outperforms a generalist Claude on security questions. Third, parallelism — multiple subagents running concurrently on independent tasks compress wall-clock time. There is a limitation: subagents only talk to the main agent, not each other (a hub-and-spoke model). For genuinely collaborative work where three specialists need to share a task list, the experimental Agent Teams feature (behind the CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS flag) gives you the tumor-board mode — a team lead spawns 3-5 subagents that operate semi-independently and report back through a shared list. Three subagents that reappear in the Software Factory capstone: codebase-researcher (Haiku, read-only — Read/Grep/Glob, returns a structured inventory of an existing codebase), story-writer (Sonnet, Read-only, transforms a rough idea plus exploration findings into a user story with acceptance criteria), and spec-writer (Sonnet, Read/Grep/Glob, drafts a technical brief from an approved story). The decision tree: single agent for one feature, subagent for self-contained tasks that produce a return value, agent team for complex multi-module builds where parallel agents need to coordinate.

## Notebook 6: CCI Session 10 - Lesson 6: Worktrees and Workflows

Worktrees and workflows are two different ideas that often get confused because they are both about doing more than one thing at once. A worktree is about isolation. A workflow is about orchestration. Worktrees first. A git worktree is a parallel working directory tied to a different branch of the same repository — like a second clone, but lighter, because the worktrees share the underlying git database. `claude --worktree feature-name` creates a worktree on the named branch and launches a Claude session in it. With three terminal panes open, you can have three Claudes working on three branches simultaneously without any one Claude touching another's files. Use cases: refactor the AKI staging logic in one worktree, fix a pathology prompt bug in another, regenerate dashboards in a third — three workflows, zero conflict. When a worktree session exits, Claude handles cleanup based on whether changes were made: no changes means the worktree is removed, changes mean it stays and you can merge the branch back to main. Worktrees can also be applied to subagents — pass `isolation: 'worktree'` and the subagent gets its own working copy automatically. Now workflows. A workflow is a scripted chain of agents, often with human approval gates between them, that turns an idea into shipped code. The Software Factory pipeline (covered in Lessons 8 and 9) is a workflow with seven stages and three approval gates. Workflows are encoded as skills — a `feature-factory` SKILL.md describes the stages, names the subagents invoked at each, and specifies the approval-gate prompts the human is asked between stages. The two concepts compose: a workflow can spawn subagents that each run in their own worktree, so you can have a backend-builder and a frontend-builder operating in parallel without stepping on each other. The medical analogy for a workflow is a clinical pathway with explicit checkpoints — initial assessment, workup, multidisciplinary discussion, treatment plan, sign-off. The pathway exists so the right pieces happen in the right order with the right reviews.

## Notebook 7: CCI Session 10 - Lesson 7: Long-Running — Loops, Schedules, Hooks

Three patterns let Claude work without you watching. First, loops. The `/loop` slash command schedules a recurring background check ("check if the deploy succeeded in 5 minutes") that Claude reports back on when news arrives. For sustained autonomous work over hours, the Ralph loop pattern uses a Stop hook to keep Claude running — every time Claude tries to exit, the hook re-injects the prompt so the work continues. Boris Cherny, the creator of Claude Code, reports that his last 30 days of development on Claude Code itself produced 259 pull requests and 457 commits, every line written by Claude with stop hooks. The danger is infinite loops — set a max_iterations and a completion_promise (an explicit criterion for what done means) or Claude will burn tokens forever. Second, schedules. A cron job that runs Claude in headless mode (`claude -p "do X" --allowedTools ...`) gives you scheduled jobs without an interactive session. The AI Office uses this for the nightly extraction pipelines: at 2 AM Claude reads the new VistA dump, runs the pathology extractor on new reports, and emails the summary stats. Headless mode is `claude` minus the conversation — single-shot, no approval prompts (so you allowlist tools narrowly up front), output to stdout. Third, hooks. A hook is a shell command that fires at a lifecycle event. Five events matter most. PreToolUse fires before any tool runs — perfect for blocking destructive commands (`rm -rf`, `DROP TABLE`, writes to PHI paths). PostToolUse fires after a tool finishes — perfect for auto-formatting after edits (Prettier or ruff). Stop fires when Claude tries to end the session — perfect for running the eval suite. SessionStart and SessionEnd fire at session boundaries. Hooks always run, with no LLM cost, on every applicable event. The principle is: CLAUDE.md for suggestions (Claude follows them about 80% of the time), hooks for requirements (they run every time, no exceptions). For clinical AI, the non-negotiable hooks are: PreToolUse blocks on destructive commands, PostToolUse formats with ruff, Stop runs the deceased-patient eval cohort. Configure them in `.claude/settings.json`.

## Notebook 8: CCI Session 10 - Lesson 8: The Software Factory, Part 1 — PRD to Story to Brief

The Software Factory is a framework adapted from the freeCodeCamp article "How to Build a Software Factory with Claude Code." It is a seven-agent pipeline with three human approval gates that turns a rough idea into shipped code. The application built in Lessons 8 and 9 is a real Django ER triage app for KHCC. The nurse screen presents a form for chief complaint, age, vitals, and oncology-specific context (known malignancy yes/no, on chemotherapy yes/no, days since last cycle, neutropenia known yes/no). On submit, the system computes an ESI-like acuity 1 to 5, runs the chief complaint through an Azure OpenAI gpt-4.1-mini extraction looking for suspected oncologic emergencies (neutropenic fever, tumor lysis syndrome, spinal cord compression, hypercalcemia of malignancy), and queues the patient. The doctor screen shows a live queue sorted by acuity, color-coded flags for the four oncologic emergencies, and a click-into-patient detail view. Lesson 8 covers the front half of the pipeline. Setup is a fresh `django-admin startproject er_triage`, a tight starter CLAUDE.md (stack, PHI rules, eval gate, test commands), and three subagents from Lesson 5 installed into `.claude/agents/`: codebase-researcher (Haiku), story-writer (Sonnet), and spec-writer (Sonnet). Stage 0 is the PRD: the student invokes the PRD-Builder skill, which classifies the project as AIDI clinical, asks one focused question about clinical service and success criterion, and drafts the full PRD with sections for Clinical Context, Users (triage nurse, ER physician, hem-onc fellow on call), Functional Requirements (acuity calculation, oncologic-emergency extraction, queue display), Non-functional Requirements (PHI handling, latency under 3 seconds, uptime), Data Model, Architecture, and a Validation Plan against a frozen deceased-patient cohort. Stage 1 is the codebase-researcher inventorying the fresh Django scaffold. Stage 2 is the story-writer producing the first user story (nurse form submission) with eight to ten acceptance criteria — this is the first approval gate. Stage 3 is the spec-writer producing the technical brief: Patient and TriageEvent models, AcuityCalculator and OncologicEmergencyExtractor services, the NurseForm view, URL routes, named test list, file plan — this is the second approval gate. The lesson ends with three approved artifacts (PRD, story, brief), the Django scaffold, the three subagents installed, and the student ready for Lesson 9's implementation half.

## Notebook 9: CCI Session 10 - Lesson 9: The Software Factory, Part 2 — Build, Test, Validate

Lesson 9 picks up from the frozen brief at the end of Lesson 8. The remaining four stages are: Stage 4 backend implementation, Stage 5 frontend implementation, Stage 6 acceptance verification, and Stage 7 implementation review. Stage 4 uses the backend-builder subagent (Sonnet, with Read, Edit, Write, Bash tool access scoped to backend folders) and a project skill called build-with-tests that codifies the process — read CLAUDE.md, match existing patterns, write code and unit tests in the same commit, run typecheck and lint and tests after every edit. The backend-builder produces: a Django migration for Patient and TriageEvent models, services/acuity.py with the rule-based ESI calculator, services/oncologic_emergency.py with the Azure OpenAI plus Pydantic-schema extractor for the four emergencies, views.py with the NurseForm view, urls.py routes, and six unit tests covering acuity edge cases, extractor schema validation, and a view smoke test. Stage 5 uses the frontend-builder subagent (Sonnet, scoped to frontend folders) to produce two Django templates (nurse_form.html with htmx for inline acuity preview, triage_confirmation.html), minimal Tailwind CSS via CDN to avoid a build step. Stage 6 uses the test-verifier subagent to write tests/test_acceptance.py — a single end-to-end test that creates a Patient, posts to /triage/new, asserts the TriageEvent was created with the correct acuity and emergency flags, satisfying the acceptance criteria from Stage 2. Stage 7 uses the implementation-validator subagent (Sonnet, read-only) to compare the code on disk against the original brief and story. The validator returns findings grouped by severity. In this capstone, a critical finding is deliberately planted: the OncologicEmergencyExtractor calls Azure OpenAI without catching network timeouts, so a transient API outage 500s the nurse form. Students see Stage 7 catch this. The pipeline loops back to Stage 4, the backend-builder adds try/except plus a graceful fallback (queue the patient with extraction status "pending"), Stages 6 and 7 re-run, and the validator approves. This is the third approval gate. After the merge, the eval-suite hook from Lesson 7 is installed into .claude/settings.json so any future code change triggers the test suite automatically. The session ends with a running Django app on local Postgres, instructions for an optional Azure App Service deploy in an appendix, and a Try This exercise: add the doctor's queue screen via the same seven-stage pipeline.

## Notebook 10: CCI Session 10 - Lesson 10: Cheat Sheet and What to Ship Next

The chain Session 10 taught is small. PRD becomes CLAUDE.md becomes skills plus subagents plus hooks becomes a worktree-parallelized build becomes an eval-gated deploy. Five wireable links. The cheat sheet reduces three hours of teaching to one page: ten slash commands to memorize (init, clear, compact, rewind, model, cost, permissions, agents, config, statusline), nine keystrokes worth knowing (Shift+Tab for Plan Mode, Ctrl+G to edit a plan, Esc to stop, Esc Esc for rewind, ! for bash output, @ for file reference, Cmd+V for screenshot paste), the standard project file layout under .claude/, a safe settings.json with allow and deny lists plus auto-format and destructive-command hooks, the starter CLAUDE.md for clinical AI work, and the seven-agent Software Factory pipeline table. Five places to take the work next, in order of effort: deploy Lesson 9's ER triage to a sandbox Azure App Service; add the doctor's queue screen through the same pipeline; wire the PRD-Builder skill into a personal project outside the AI Office; replace one Excel workflow with a Claude Code script; and start building your own subagent library — a methods-section-writer for manuscripts, a cohort-sql-reviewer for AIDI pipelines, a consent-form-translator for IRB submissions. The two-line summary of the session: students learned how to turn a clinical idea into a PRD, the PRD into a CLAUDE.md, the CLAUDE.md into skills and subagents, the skills and subagents into a pipeline, and the pipeline into a shipped application — the discipline is small; the leverage is enormous. The closing warning: do not ship anything with a clinical decision component without the eval discipline — a frozen deceased-patient test cohort and a binary regression gate. The Software Factory gives speed. The eval cohort gives safety. They are not interchangeable.
