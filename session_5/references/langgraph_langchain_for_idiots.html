0APlain Language Glossary

Think of this as the "decoder ring" for vocabulary that shows up in docs, error messages, and the chapters below. You do not need to memorise definitions — you need a mental picture so the code snippets stop feeling like random spells.

Runnable

In LangChain, a Runnable is any building block that follows the same simple contract: you hand it input, you get output. The magic word for "run it" is .invoke(...) (batch and stream variants exist, but invoke is the one you will see first).

Clinical analogy: a Runnable is like a orderable clinical module — "CBC interpretation", "drug interaction check", "summarise discharge note" — each module knows how to accept a standard requisition and return a standard result. LangChain wires those modules in sequence because they all share the same plug shape.

A chat model, a prompt template, a retriever, an agent, and even an entire LangGraph CompiledStateGraph can all behave as Runnables. That is why you keep seeing model.invoke(...), retriever.invoke(...), and agent.invoke(...) — same verb, different "specialty."

invoke() and RunnableConfig

invoke means "execute one logical step with this payload." For agents and graphs, the payload is usually a dictionary (for example {"messages": [...]}). For a bare model it might just be a string or a list of messages.

RunnableConfig is the optional second argument carrying runtime settings — not part of the medical chart, but part of how the run is wired: which conversation thread (thread_id), tracing tags, recursion limits, and other tunables. Chapter 7 shows def node(state, config) when a graph node needs to read those settings (for example to log the active thread_id).

Versioning, semver, and "pinning"

Python libraries publish version numbers so you know what code you installed. LangChain splits functionality across several packages (langchain, langchain-core, langgraph, provider packages like langchain-openai, etc.). They do not all move in lockstep.

Semver (semantic versioning) is a polite promise about compatibility: versions look like MAJOR.MINOR.PATCH (for example 1.2.3). When the major number changes, the authors are warning you that breaking changes are allowed. When only minor or patch changes, upgrades are meant to be gentler.

Pinning means writing your requirements.txt so pip installs a known safe band, e.g. langchain>=1.0,<2.0 ("1.x series only"). That prevents a silent pip install --upgrade from jumping you to a future 2.0 that renames half the imports overnight.

Why the book fusses about langchain-community: that mega-package historically did not obey semver strictly. Treat it like blood products — only upgrade on purpose, pin to a minor range, and prefer smaller dedicated packages when they exist. Chapter 3 lists concrete pins.

Middleware (agent middleware)

Middleware is code that sits around the model step of an agent — not replacing the model, but intercepting the loop so you can add cross-cutting behaviour: logging, guardrails, retries, human approval, token trimming, and so on.

Clinical analogy: middleware is like the safety checks around high-risk orders: pharmacy verification, dual sign-off for chemotherapy, or an EHR alert that fires before the order is committed. The clinician (the LLM) still writes the order, but the workflow can pause, ask questions, or attach mandatory context.

LangChain 1.0 agents expect you to attach middleware as a list on create_agent(..., middleware=[...]). Hooks such as before_model / after_model are just named interception points — "run this snippet before the LLM is called" or "right after the LLM responds." Chapter 5 walks through human-in-the-loop middleware in detail.

Imagine a three-layer cake. At the bottom is LangChain — the foundation: models, tools, prompts. In the middle is LangGraph — the wiring you use to build workflows with loops, branches, and state. On top is Deep Agents — a batteries-included kit that gives you planning, file management, subagents, and memory for free.

Deep Agents
Highest level. Batteries included. Planning, skills, memory, files, subagents, human-in-the-loop — all configured, not hand-coded.
LangGraph
Middle level. You own the control flow. Nodes, edges, state, checkpointers. You decide every step and every branch.
LangChain
Foundation. The building blocks. Models, tools, prompt templates, retrievers, create_agent().
📝
The Layers Cooperate

Using Deep Agents doesn't mean abandoning LangChain — your tools, models, and retrievers are still LangChain objects. Using LangGraph doesn't lock out LangChain either; nodes call LangChain models and tools freely. They stack, they don't compete.

The Decision Ladder

Walk down this list. Stop at the first yes.

Ask yourself…If yes →
Does the task need planning, file management, persistent memory, or on-demand skills?Deep Agents
Does the task need loops, branching, parallel workers, or custom human-in-the-loop?LangGraph
Is this a single-purpose agent that takes input, runs tools, and returns a result?LangChain (create_agent)
Is this a pure model call or retrieval pipeline with no agent loop?LangChain (chain)

Profile: LangChain

Use when: single-purpose agent, RAG pipeline, or prompt-chain style work.
Skip when: you need many coordinated steps, branching, or memory that survives across sessions.

Profile: LangGraph

Use when: you want total control — retry-until-correct loops, reflection, map-reduce fan-out, approval gates at specific steps.
Skip when: you want planning and file management handed to you (use Deep Agents).

Profile: Deep Agents

Use when: a long-running, open-ended task needs a todo list, a virtual filesystem, subagents, and skills that load on demand.
Skip when: the job is a single tool call, or you need hand-tuned graph edges.

🧠
Deep Agents Middleware Stack

Deep Agents ships a built-in middleware layer — you configure, not implement. If the word middleware still sounds like enterprise IT, read Glossary → Middleware first; then the list below will read like a stack of optional safety rails rather than random buzzwords.

  • TodoListMiddleware — write_todos tool (always on)
  • FilesystemMiddleware — ls, read_file, write_file, edit_file, glob, grep (always on)
  • SubAgentMiddleware — task tool to spawn subagents (always on)
  • SkillsMiddleware — load SKILL.md files on demand (opt-in)
  • MemoryMiddleware — cross-session memory via a Store (opt-in)
  • HumanInTheLoopMiddleware — approval gates before sensitive tool calls (opt-in)

Why everything is .invoke(...)

Under the hood, an agent built with create_agent is a Runnable: the same "give input, get output" contract as a model or a retriever. That is why the examples all end with agent.invoke({"messages": ...}). If the word Runnable makes you twitch, read Glossary → Runnable — it is shorter than it sounds.

📝
What "middleware" means in this row

Here it is not hospital middleware in the IT sense. It is a list of small functions LangChain calls before or after the LLM thinks, so you can pause for approval, trim context, log tool calls, or retry failures without rewriting the whole agent. Same idea as double-checks in a CPOE workflow. Full story: Glossary → Middleware and Chapter 5.

Your First Agent

from langchain.agents import create_agent
from langchain_core.tools import tool

@tool
def get_weather(location: str) -> str:
    """Get current weather for a location.

    Args:
        location: City name
    """
    return f"Weather in {location}: Sunny, 72F"

agent = create_agent(
    model="anthropic:claude-sonnet-4-5",
    tools=[get_weather],
    system_prompt="You are a helpful assistant."
)

result = agent.invoke({
    "messages": [{"role": "user", "content": "What's the weather in Paris?"}]
})
print(result["messages"][-1].content)

Give It a Memory

from langchain.agents import create_agent
from langgraph.checkpoint.memory import MemorySaver

checkpointer = MemorySaver()

agent = create_agent(
    model="anthropic:claude-sonnet-4-5",
    tools=[search],
    checkpointer=checkpointer,
)

config = {"configurable": {"thread_id": "user-123"}}
agent.invoke({"messages": [{"role": "user", "content": "My name is Alice"}]}, config=config)
result = agent.invoke({"messages": [{"role": "user", "content": "What's my name?"}]}, config=config)
# Agent remembers: "Your name is Alice"
💡
thread_id = the Conversation

Without a thread_id, each invoke is a goldfish — zero memory. Pass the same thread_id to stay in the same conversation. Give Bob a different thread_id from Alice and they won't see each other's history.

The LLM only sees the docstring and argument descriptions to decide when to call a tool. "Does stuff" is not going to cut it.

Structured Output with Pydantic

from langchain.agents import create_agent
from pydantic import BaseModel, Field

class ContactInfo(BaseModel):
    name: str
    email: str
    phone: str = Field(description="Phone number with area code")

# Option 1: agent-level structured output
agent = create_agent(model="gpt-4.1", tools=[search], response_format=ContactInfo)
result = agent.invoke({"messages": [{"role": "user", "content": "Find contact for John"}]})
print(result["structured_response"])  # ContactInfo(name='John', ...)

# Option 2: model-level structured output (no agent needed)
from langchain_openai import ChatOpenAI
model = ChatOpenAI(model="gpt-4.1")
structured_model = model.with_structured_output(ContactInfo)
response = structured_model.invoke("Extract: John, john@example.com, 555-1234")
# ContactInfo(name='John', email='john@example.com', phone='555-1234')

DON'T

DON'T

    {"messages": [("user", "Do research")]},

Ch 5Middleware & Human-in-the-Loop

In This Chapter

  • Pausing your agent for human approval
  • The three decision types: approve, edit, reject

Middleware is the seatbelt of your agent. It intercepts the agent loop so you can add human approval, retry logic, logging, or anything else — without rewriting the agent itself.

More slowly: picture each turn of the agent as read chart → think → maybe call a tool → think again. Middleware lets you hang tiny programmes on those transitions. Some run before the model sees the chart (for example stripping identifiers), some after the model drafts an action (for example forcing a human "sign" before send_email fires). LangChain exposes those moments as named hooks so you are not copy-pasting the same guard code into every project.

💡
If you only remember one sentence

Middleware = "extra clinicians in the corridor" who can slow the case down, add warnings, or demand co-signatures — the attending LLM still makes the plan, but the corridor staff keep the process safe.

The HITL Setup

Always needs: a checkpointer + a thread_id. No exceptions.

from langchain.agents import create_agent
from langchain.agents.middleware import HumanInTheLoopMiddleware
from langgraph.checkpoint.memory import MemorySaver
from langchain.tools import tool

@tool
def send_email(to: str, subject: str, body: str) -> str:
    """Send an email."""
    return f"Email sent to {to}"

agent = create_agent(
    model="gpt-4.1",
    tools=[send_email],
    checkpointer=MemorySaver(),  # Required for HITL
    middleware=[
        HumanInTheLoopMiddleware(
            interrupt_on={
                "send_email": {"allowed_decisions": ["approve", "edit", "reject"]},
            }
        )
    ],
)

Running With Interrupts

from langgraph.types import Command

config = {"configurable": {"thread_id": "session-1"}}

# Step 1: Agent runs until it wants to call the tool
result1 = agent.invoke({
    "messages": [{"role": "user", "content": "Send email to john@example.com"}]
}, config=config)

# Step 2: Detect the pause
if "__interrupt__" in result1:
    print(f"Waiting for approval: {result1['__interrupt__']}")

# Step 3: Human approves
result2 = agent.invoke(
    Command(resume={"decisions": [{"type": "approve"}]}),
    config=config
)

The Three Decisions

Approve

agent.invoke(Command(resume={"decisions": [{"type": "approve"}]}), config=config)

Edit the arguments before the tool runs

agent.invoke(
    Command(resume={
        "decisions": [{
            "type": "edit",
            "edited_action": {
                "name": "send_email",
                "args": {
                    "to": "alice@company.com",   # fixed email
                    "subject": "Project Meeting - Updated",
                    "body": "...",
                },
            },
        }]
    }),
    config=config
)

Reject with feedback

agent.invoke(
    Command(resume={
        "decisions": [{
            "type": "reject",
            "feedback": "Cannot delete customer data without manager approval",
        }]
    }),
    config=config
)

Different Policies per Tool

agent = create_agent(
    model="gpt-4.1",
    tools=[send_email, read_email, delete_email],
    checkpointer=MemorySaver(),
    middleware=[
        HumanInTheLoopMiddleware(
            interrupt_on={
                "send_email":  {"allowed_decisions": ["approve", "edit", "reject"]},
                "delete_email": {"allowed_decisions": ["approve", "reject"]},  # no edit
                "read_email":  False,  # no HITL for reading
            }
        )
    ],
)

Custom Middleware Hooks

Two hook patterns:

  • Wrap hooks — wrap_tool_call, wrap_model_call: signature (request, handler). Call handler(request) to proceed, or return early to short-circuit.
  • Before/after hooks — before_model, after_model, before_agent, after_agent: signature (state, runtime). Inspect / modify state. Return None or a dict of updates.
⚠
Don't yield in @wrap_tool_call

Using yield makes the hook a generator and throws NotImplementedError. Always return handler(request).

from langchain.agents.middleware import wrap_tool_call

@wrap_tool_call
def retry_middleware(request, handler):
    for attempt in range(3):
        try:
            return handler(request)
        except Exception:
            if attempt == 2:
                raise

@wrap_tool_call
def guard_middleware(request, handler):
    if request.tool_call["name"] == "dangerous_tool":
        return "This tool is disabled"  # short-circuit
    return handler(request)
from langchain.agents.middleware import before_model, after_model

@before_model
def log_calls(state, runtime):
    print(f"Calling model with {len(state['messages'])} messages")

@after_model
def check_output(state, runtime):
    print("Model responded")

RAG in 30 Seconds

The LLM doesn't know your documents. So: chop them up, turn each chunk into a vector, store those vectors. At query time, vectorize the question, find the closest chunks, stuff them into the prompt, and let the LLM answer.

The Pipeline

  1. Index — load → split → embed → store
  2. Retrieve — query → embed → search → return docs
  3. Generate — docs + query → LLM → response

Ch 7StateGraph Fundamentals

In This Chapter

  • State, nodes, edges — the holy trinity

LangGraph models your workflow as a directed graph. You describe shared state, write nodes that update state, and connect them with edges. You must compile() the graph before running it.

The 5-Step Design Methodology

  1. Map discrete steps — sketch a flowchart; each step becomes a node.
  2. Identify what each step does — LLM step? data step? action step? user input?
  3. Design your state — store raw data, not formatted prompts.
  4. Build your nodes — each returns a partial update dict.
  5. Wire it together — add edges, conditional routing, compile with a checkpointer if needed.

from langgraph.graph import StateGraph, START, END
from typing_extensions import TypedDict

class State(TypedDict):
    input: str
    output: str

def process_input(state: State) -> dict:
    return {"output": f"Processed: {state['input']}"}

def finalize(state: State) -> dict:
    return {"output": state["output"].upper()}

graph = (
    StateGraph(State)
    .add_node("process", process_input)
    .add_node("finalize", finalize)
    .add_edge(START, "process")
    .add_edge("process", "finalize")
    .add_edge("finalize", END)
    .compile()
)

Conditional routing

from typing import Literal
from langgraph.graph import StateGraph, START, END

class State(TypedDict):
    query: str
    route: str
    result: str

def classify(state: State) -> dict:
    if "weather" in state["query"].lower():
        return {"route": "weather"}
    return {"route": "general"}

def route_query(state: State) -> Literal["weather", "general"]:
    return state["route"]

graph = (
    StateGraph(State)
    .add_node("classify", classify)
    .add_node("weather", lambda s: {"result": "Sunny, 72F"})
    .add_node("general", lambda s: {"result": "General response"})
    .add_edge(START, "classify")
    .add_conditional_edges("classify", route_query, ["weather", "general"])
    .add_edge("weather", END)
    .add_edge("general", END)
    .compile()
)

Command: Update State AND Route

Command has three fields:

  • update — state updates to apply (like a node's return dict)
  • goto — next node name(s)
  • resume — for resuming after an interrupt() (Ch 8)
from langgraph.types import Command
from typing import Literal

class State(TypedDict):
    count: int
    result: str

def node_a(state: State) -> Command[Literal["node_b", "node_c"]]:
    new_count = state["count"] + 1
    if new_count > 5:
        return Command(update={"count": new_count}, goto="node_c")
    return Command(update={"count": new_count}, goto="node_b")
⚠
Static edges still run

Command only adds a dynamic edge. If node_a returns Command(goto="node_c") and you also wrote graph.add_edge("node_a", "node_b"), both node_b and node_c will run.

Send: Orchestrator / Worker Fan-Out

from langgraph.types import Send
from typing import Annotated
import operator

class OrchestratorState(TypedDict):
    tasks: list[str]
    results: Annotated[list, operator.add]   # reducer REQUIRED
    summary: str

def orchestrator(state: OrchestratorState):
    return [Send("worker", {"task": task}) for task in state["tasks"]]

def worker(state: dict) -> dict:
    return {"results": [f"Completed: {state['task']}"]}

def synthesize(state: OrchestratorState) -> dict:
    return {"summary": f"Processed {len(state['results'])} tasks"}

graph = (
    StateGraph(OrchestratorState)
    .add_node("worker", worker)
    .add_node("synthesize", synthesize)
    .add_conditional_edges(START, orchestrator, ["worker"])
    .add_edge("worker", "synthesize")
    .add_edge("synthesize", END)
    .compile()
)

result = graph.invoke({"tasks": ["Task A", "Task B", "Task C"]})

Without Annotated[list, operator.add] on the results field, only the last worker's output survives.

Approval Workflow with Routing

from langgraph.types import interrupt, Command
from langgraph.graph import END
from typing import Literal

class EmailAgentState(TypedDict):
    email_content: str
    draft_response: str
    classification: dict

def human_review(state: EmailAgentState) -> Command[Literal["send_reply", "__end__"]]:
    classification = state.get("classification", {})

    # interrupt() must come first — any code before it will re-run on resume
    human_decision = interrupt({
        "email_id": state.get("email_content", ""),
        "draft_response": state.get("draft_response", ""),
        "urgency": classification.get("urgency"),
        "action": "Please review and approve/edit this response",
    })

    if human_decision.get("approved"):
        return Command(
            update={"draft_response": human_decision.get("edited_response", state.get("draft_response", ""))},
            goto="send_reply",
        )
    else:
        return Command(update={}, goto=END)

The Idempotency Rule

⚠
Code Before interrupt() Re-Runs

When the graph resumes, the entire node runs again from the top. Any side effect you did before interrupt() — DB inserts, list appends, emails sent — will happen a second time.

Rules of thumb

  • Do use upsert (not insert) before interrupt().
  • Do put side effects after the interrupt if you can.
  • Do split side effects into their own nodes.
  • Don't create new records before interrupt().
  • Don't append to lists before interrupt().
DON'T 
def node_a(state: State):
    audit_id = db.create_audit_log({  # duplicates on each resume!
        "user_id": state["user_id"],
        "action": "pending_approval",
    })
    approved = interrupt("Approve this change?")
    return {"approved": approved}
DO 
def node_a(state: State):
    # Upsert is idempotent — safe before interrupt
    db.upsert_user(user_id=state["user_id"], status="pending_approval")
    approved = interrupt("Approve this change?")
    return {"approved": approved}

# OR: put the side effect AFTER interrupt
def node_a(state: State):
    approved = interrupt("Approve this change?")
    if approved:
        db.create_audit_log(user_id=state["user_id"], action="approved")
    return {"approved": approved}
🧠
Subgraphs Amplify This

If a subgraph contains an interrupt(), resuming re-executes both the parent node and the subgraph node. Stay idempotent at every layer.

When to use each

  • checkpointer=False — subgraph needs nothing extra. Simplest, least overhead.
  • default (omit) — subgraph needs interrupt() but not cross-invocation memory. Parallel calls work.
  • checkpointer=True — subgraph needs to remember across invocations (multi-turn conversation).
⚠
Stateful subgraphs can't be called twice in parallel

Two calls to the same checkpointer=True subgraph inside one node fight for the same namespace. Wrap each in its own StateGraph with a unique node name to keep namespaces stable.

        .add_edge("__start__", name)

Hybrid Routing with CompositeBackend

Security

Never use FilesystemBackend in a web server. An LLM with real disk write access + any prompt injection = disaster. Use StateBackend or a sandbox.

Using the Store from Custom Tools

from langchain.tools import tool, ToolRuntime
from langchain.agents import create_agent
from langgraph.store.memory import InMemoryStore

@tool
def get_user_preference(key: str, runtime: ToolRuntime) -> str:
    """Get a user preference from long-term storage."""
    store = runtime.store
    result = store.get(("user_prefs",), key)
    return str(result.value) if result else "Not found"

@tool
def save_user_preference(key: str, value: str, runtime: ToolRuntime) -> str:
    """Save a user preference to long-term storage."""
    store = runtime.store
    store.put(("user_prefs",), key, {"value": value})
    return f"Saved {key}={value}"

store = InMemoryStore()

agent = create_agent(
    model="gpt-4.1",
    tools=[get_user_preference, save_user_preference],
    store=store,
)

Ch 12Subagents, Todos, and HITL

In This Chapter

  • Delegating work with the task tool
  • Automatic task planning with write_todos

Deep Agents ships with three orchestration capabilities, automatic on every agent: SubAgentMiddleware for delegation, TodoListMiddleware for planning, and HumanInTheLoopMiddleware for approval gates.

Subagents

Use subagents when…Use the main agent when…
Task needs specialized toolsGeneral-purpose tools are enough
You want to isolate complex workSingle-step operation
Need clean context for the main agentContext bloat is acceptable

How it works

Main agent has the task tool → creates a fresh subagent → subagent executes autonomously with its own tools and context → returns a final report. The main agent sees only the report.

📝
The Default "general-purpose" Subagent

You always have a general-purpose subagent available — it inherits the main agent's tools, model, and config. Custom subagents don't inherit skills; you declare them explicitly.

Full approval workflow

from langgraph.types import Command

agent = create_deep_agent(
    interrupt_on={"write_file": True},
    checkpointer=MemorySaver(),
)

config = {"configurable": {"thread_id": "session-1"}}

# Step 1: agent proposes write_file; execution pauses
result = agent.invoke({
    "messages": [{"role": "user", "content": "Write config to /prod.yaml"}]
}, config=config)

# Step 2: check for interrupts
state = agent.get_state(config)
if state.next:
    print("Pending action")

# Step 3: approve and resume
result = agent.invoke(
    Command(resume={"decisions": [{"type": "approve"}]}),
    config=config,
)
# Reject with feedback
agent.invoke(
    Command(resume={"decisions": [{
        "type": "reject",
        "message": "Run tests first",
    }]}),
    config=config,
)
# Edit before execution
agent.invoke(
    Command(resume={"decisions": [{
        "type": "edit",
        "edited_action": {
            "name": "execute_sql",
            "args": {"query": "DELETE FROM users WHERE last_login < '2020-01-01' LIMIT 100"},
        },
    }]}),
    config=config,
)

Always / Never Rules

  • Always pass thread_id when you want memory or interrupts.
  • Always use a checkpointer with interrupt_on / HITL.
  • Always compile() before calling invoke.
  • Always annotate list fields with a reducer if two nodes might write them.
  • Never mutate state inside a node — return a partial update dict.
  • Never use InMemorySaver or InMemoryStore in production.
  • Never insert rows or append to lists before an interrupt() — it'll re-run.
  • Never route back to START.
  • Never give FilesystemBackend disk access in a public web server.

Stream Modes at a Glance

  • values — full state after each step (monitoring)
  • updates — just the deltas
  • messages — LLM tokens (chat UIs)
  • custom — your progress strings via get_stream_writer()
