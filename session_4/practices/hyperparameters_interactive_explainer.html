Hyperparameters, Explained

When you fine-tune a model you don't change its architecture — you change how it learns. These settings (hyperparameters) control speed, stability, and quality. Move the sliders below to build intuition before your Colab lab.

⚙

Learning Rate

The learning rate (LR) is the single most important hyperparameter. It controls how much the model's weights change after each batch of examples. For transformer fine-tuning the sweet spot is usually 2 × 10-5 to 5 × 10-5.

🚗
Analogy — driving speed. Imagine driving toward a parking spot. A huge LR is like flooring the gas — you overshoot and crash (loss explodes). A tiny LR is like crawling at 1 km/h — you get there eventually, but it takes forever. The right LR gets you there smoothly.
Drag to change LR (log scale) 3.2e-5
📦

Batch Size

Batch size is how many examples the model sees before updating its weights. A batch of 1 updates after every single example (very noisy). A batch of 32 averages gradients across 32 examples (much smoother, but uses more GPU memory).

🎓
Analogy — studying for an exam. Batch = 1 is like changing your study strategy after reading one flashcard — reactive but erratic. Batch = 32 is like reviewing a whole chapter first, then deciding what to change — slower per-decision but more reliable.
Batch size 4
🔁

Epochs & Overfitting

One epoch = one complete pass through the entire training set. More epochs give the model more chances to learn. But after a point the model starts memorizing the training data instead of learning general patterns — that is overfitting.

📖
Analogy — re-reading a textbook. Reading it 3 times helps you understand concepts (generalization). Reading it 30 times means you memorize page numbers instead of ideas (overfitting). The validation curve tells you when you have crossed the line.
Number of epochs 15
📈

Warmup & Cosine Decay

Instead of using one fixed LR, most fine-tunes ramp up slowly (warmup), reach a peak, then gradually decrease (decay). This protects the pretrained weights from large early updates and lets the model settle gently at the end.

🏋
Analogy — a workout. You warm up with light stretching (small LR), hit peak intensity mid-session (full LR), then cool down (decaying LR). Skipping the warmup risks injury (destroying pretrained knowledge).
Warmup (% of total steps) 8%
In HuggingFace: warmup_ratio=0.08 and lr_scheduler_type="cosine" in your TrainingArguments.
📚

Theory Cheat-Sheet

◆ Gradient descent

At each step the model measures how wrong it was (loss), computes which direction to adjust each weight (gradient), then nudges the weights in that direction. The learning rate controls the size of the nudge.

◆ Adam / AdamW optimizer

Instead of using the same step size for every weight, Adam keeps a running average of each weight's gradients and adapts. AdamW adds weight decay (a gentle pull toward zero) that is decoupled from the LR — the standard optimizer for transformer fine-tuning.

◆ Gradient accumulation

If your GPU can only fit batch size 2, but you want the effect of batch 16: run 8 forward passes, accumulate the gradients, then do one update. In HuggingFace: gradient_accumulation_steps=8. Same math, less memory.

◆ Catastrophic forgetting

A pretrained model already "knows" language. If your LR is too high, the first few updates overwrite that knowledge before the model can adapt it to your task. This is why fine-tuning LRs (2e-5) are 100× smaller than pretraining LRs (1e-3).

◆ Early stopping

Monitor validation loss every epoch. When it stops improving for N epochs in a row ("patience"), stop training and keep the best checkpoint. Prevents wasting time and avoids overfitting.

◆ Typical HuggingFace config

learning_rate=2e-5, per_device_train_batch_size=4, num_train_epochs=5, warmup_ratio=0.06, lr_scheduler_type="cosine", weight_decay=0.01, evaluation_strategy="epoch", save_strategy="epoch", load_best_model_at_end=True.

KHCC & Clinical Tips

  • Data stays local. Run fine-tuning in your own Colab runtime. De-identify reports before uploading. Never log PHI to public dashboards (e.g. W&B public mode).
  • Always use a GPU. Go to Runtime → Change runtime type → T4 GPU in Colab before starting training. CPU fine-tuning is 50-100× slower.
  • Save checkpoints. Use save_strategy="epoch" so you can go back to the best epoch if later ones overfit.
  • Document everything. Record your exact LR, batch size, epochs, and ROUGE scores in your notebook. Reproducibility is required for clinical audit trails.
  • Watch both curves. If training loss drops but validation loss rises, you are overfitting. Stop earlier or add regularization (weight decay, dropout).
  • Start conservative. Begin with lr=2e-5, batch=4, epochs=5. Only change one thing at a time so you can see what helps.
