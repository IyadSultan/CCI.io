Move the sliders. Watch toy training curves and schedules change. This is a teaching simulation — not a real trainer — but it matches the shape of what you see when fine-tuning radiology summarization in Colab.

Learning rate (step size each update)

Transformer fine-tunes often use 2e-5 to 5e-5 as a starting point. Too large → unstable loss or forgetting pretrained weights. Too small → barely moves.

lr = 3.2e-5

Batch size (examples per update)

Larger batches give smoother gradient estimates; tiny batches are noisier but can help escape shallow minima. GPU memory caps how large you can go.

batch = 4

Epochs & the validation curve

One epoch = one full pass over the training set. Training loss usually keeps dropping. Validation loss often bottoms out then rises — that gap is overfitting.

epochs = 18

Warmup + cosine decay (typical schedule)

Many HF configs ramp LR up for the first few hundred steps (warmup), then decay toward zero. That avoids huge early updates right after loading pretrained weights.

warmup = 8%
The curve shows multiplier × peak LR over training steps. Your HuggingFace TrainingArguments can set warmup_ratio or warmup_steps plus lr_scheduler_type="cosine".

Quick reference

Gradient descent intuition

The model’s weights are nudged in the direction that reduces loss. LR controls how big each nudge is. Adam/AdamW adapts per-parameter step sizes using moving averages of gradients.

Weight decay

A regularizer that slightly pulls weights toward zero each step. Helps generalization. In AdamW, decay is decoupled from the adaptive LR (cleaner than old L2-in-Adam).

Gradient accumulation

If batch size is 2 but you want effective batch 16, accumulate gradients over 8 forward passes before one optimizer step. Same math as a big batch, less VRAM.

Why summarization is sensitive

Radiology “findings → impression” is a seq2seq task. Small data + wrong LR often shows as ROUGE improving on train but flat or worse on validation — always watch both curves.

At KHCC-scale

  • Keep de-identified reports in the Colab runtime you control; do not mix PHI with public logging.
  • Use a GPU (T4/L4) for Trainer runs; CPU fine-tunes on transformer heads are painfully slow.
  • Save checkpoints and note learning rate, per-device batch size, and num_train_epochs in your lab write-up — reproducibility matters for audits.
